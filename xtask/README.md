# xtask

Root `cargo xtask` dispatcher for romanian-folk-fight. Owns command dispatch,
shared child-process/error/timing conventions, and the native verification
commands below. It is a workspace member (`Cargo.toml`'s `[workspace]`), never
a dependency of the game binary.

## Wiring: the `cargo xtask` alias

`cargo xtask <args>` works because `.cargo/config.toml` at the workspace root
defines `[alias] xtask = "run --package xtask --"`. That file is
gitignored and worktree-local, generated by `scripts/bootstrap-worktree.sh`
(it also pins `build.target-dir` and, when available, an `sccache`
`rustc-wrapper` for that worktree). Cargo only reads one `.cargo/config.toml`
per directory, and the workspace root has no ancestor directory inside this
repo to host a second, committed config file -- so the alias is generated
into that same file rather than living in its own location. Bootstrap
regenerates the whole file deterministically every run, so adding the alias
did not change its idempotency guarantee.

**Known limitation:** until `scripts/bootstrap-worktree.sh` has run at least
once in a given checkout, the `cargo xtask` alias does not exist yet. Either
run bootstrap first (already required for a fresh worktree, see the repo's
`AGENTS.md`/`rust-dev` skill), or invoke the crate directly:
`cargo run --package xtask -- <args>`.

## Commands

```
cargo xtask --help              # lists every root-owned command
cargo xtask test logic          # fast pure-logic unit tests
cargo xtask test journey        # one headless multi-step GameState journey
cargo xtask check build-matrix  # native + release + wasm cargo check, no `dev` leakage
cargo xtask pre-push            # fmt check, clippy, cargo test, build-matrix -- stops at first failure
```

### `test logic`

Runs `cargo test --lib` filtered to the pure game-rule modules that don't
spin up a Bevy `App`/`MinimalPlugins` at all: `character::`, `combat::ai::`,
`combat::engine::`, `creation::draft::`, `items::`, `progression::level::`,
`roster::`. These cover damage/hit/crit formulas, AI choice, the character
creation draft, item catalog/equipment, the level curve, and ladder scaling
-- the fastest meaningful subset of the suite, and a tight loop for
iterating on game-rule changes without paying for the ~250 remaining tests
that each boot a headless Bevy `App`.

### `test journey`

Targets the closest existing headless "journey" test in the repo:
`progression::tests::lifetime_earnings_accumulate_across_wins_and_reset_with_the_run`.
It drives the `GameState` machine through `Fight -> FightResult` (payout +
XP credited) `-> Fight -> FightResult` (payout accumulates) `-> GameOver`,
then presses the game-over screen's real "back to menu" button
(`Interaction::Pressed` on the actual UI entity) and asserts the run resets.

It does **not** yet cover `MainMenu`/`CharacterCreation`/`Shop` -- the
tracker (#151, via #142's children) plans a full
`menu -> creation -> fight -> result/shop -> next fight -> defeat -> reset`
journey. That flow doesn't exist yet, so this command targets the closest
true multi-step, headless, state-driven test that does, and should be
repointed at the full journey once #142 lands it.

### `check build-matrix`

Runs three `cargo check` invocations in order, stopping at the first
failure: plain native, `--release`, and `--target wasm32-unknown-unknown`
(run `rustup target add wasm32-unknown-unknown` once if it isn't installed).
None of the three ever passes `--features dev`: `dev` (Bevy dynamic linking)
exists only for fast local `cargo run --features dev` iteration and must
never leak into a release or wasm artifact. Omitting the flag *is* the
guarantee -- the root `Cargo.toml` has no default feature list that could
pull `dev` in implicitly, so the transcript never mentions it.

### `pre-push`

The full repository-native gate, in order, stopping at the first failure:
`cargo fmt --all -- --check`, `cargo clippy --all-targets -- -D warnings`,
`cargo test`, then `check build-matrix`. This mirrors what the git
`pre-push` hook and CI expect a clean tree to pass.

## Process/result conventions

Every step (every `cargo` invocation any command runs) goes through
[`process::run_step`](src/process.rs), which:

- Prints the exact command line before running it.
- Times it and reports elapsed seconds.
- Retains its combined stdout/stderr on disk (see "Artifact directory"
  below) regardless of outcome.
- On success, returns a `StepReport { label, elapsed, artifact }`.
- On failure, returns a `StepError::Failed { label, elapsed, exit_code,
  artifact }` (or `StepError::Spawn` if the command couldn't even launch),
  and echoes the captured output immediately so the failure is visible
  without opening the log file.

Commands that run multiple steps (`build-matrix`, `pre-push`) chain them with
`?`, so the first `StepError` short-circuits the rest and propagates up to
`main`, which prints `cargo xtask: stopped at first failure -> <StepError>`
and exits non-zero. `main` also handles `DispatchError::Usage` (bad
command/subcommand) separately from a step actually failing.

### Artifact directory convention

`target/xtask-artifacts/<slugified-step-label>.log` -- one file per step
(e.g. `check-native-no-dev-feature.log`, `test-logic.log`,
`fmt-check.log`), each holding the exact command line plus its captured
stdout and stderr. Files are overwritten the next time that step runs;
nothing is cleaned up automatically, so the most recent run of every step
that has ever executed in this worktree is always on disk. `run_step` prints
the path on both success and failure.

## Extension pattern for later, independently owned modules

`xtask` is root-owned (#152/#163). Later work adds its own command group as
a new module -- #141 for `asset`, #144 for `browser-smoke` -- without
editing any existing feature module (`test_cmd.rs`, `check_cmd.rs`,
`pre_push.rs`):

1. Add a new file under `xtask/src/commands/`, e.g. `assets_cmd.rs`.
2. In it, expose the same shape every group exposes:
   - `pub const ABOUT: &str`
   - `pub const SUBCOMMANDS: &[(&str, &str)]` (name + one-line help)
   - `pub fn run(sub: &str) -> Result<(), StepError>`, building each
     subcommand's `Command` and running it through
     `crate::process::run_step`, exactly like `test_cmd`/`check_cmd` do.
3. In `xtask/src/commands/mod.rs`, add `pub mod assets_cmd;` and exactly one
   entry to the `GROUPS` slice. That is the only change outside the new
   module's own file -- no other command group's file is touched, and
   `cargo xtask --help` picks the new group up automatically.

A standalone command with no subcommands (like `pre-push`) follows the same
shape minus the subcommand list: `pub const ABOUT`, `pub fn run() ->
Result<(), StepError>`, and one entry in `ROOT_COMMANDS` instead of `GROUPS`.

This module deliberately contains no placeholder `asset`/`browser-smoke`
modules or subcommands -- those are #141's and #144's to add.
